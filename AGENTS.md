# AGENTS.md — как вести документацию и собирать changelog (WildcoreDMS wiki)

Заметка для будущих агентских сессий. Описывает структуру вики, как определять и
писать changelog новой версии, правила формулировок и git-workflow.

> WildcoreDMS — **проект с закрытым исходным кодом**. В публичной документации
> НЕ раскрываем внутреннюю реализацию. См. раздел «Правила формулировок».

---

## 1. Структура репозитория

- Репозиторий вики: `github.com/wildcore-dms/wiki`, основная ветка — `master`
  (коммитим прямо в `master`, PR не заведено).
- Документация двуязычная, собирается **mkdocs-material**:
  - `uk/docs/...` — украинская версия (`uk/mkdocs.yml`)
  - `en/docs/...` — английская версия (`en/mkdocs.yml`)
  - Любую правку делаем **синхронно в обоих языках** (uk + en).
- Файлы вне `docs/` (этот `AGENTS.md`, `scrabber.php`, `providers*.csv`,
  `diagrams/` и т.п.) **в сайт не попадают** — это служебные/рабочие файлы.
  При коммите их не трогаем.

### Где живёт changelog
- `*/docs/changelog.md` — **индекс** версий: сворачиваемые блоки `note` (admonition).
  - Самая новая версия — `???+ note "X (MM.YYYY)"` (раскрыта по умолчанию).
  - Предыдущие — `??? note "..."` (свёрнуты). При добавлении новой версии
    **понижаем** предыдущую верхнюю с `???+` до `???`.
  - Внутри: `**Ключові зміни** 🔑` / `**Key changes** 🔑`, 4-6 буллетов с emoji,
    в конце ссылка `[Більше...](./changelogs/X.md)` / `[More...](./changelogs/X.md)`.
- `*/docs/changelogs/X.md` — **детальная** страница версии.
  - Заголовок `# Версія X` / `# Version X`, строкой ниже `_Дата релізу: YYYY-MM-DD_`
    / `_Release date: YYYY-MM-DD_`.
  - Разделы с emoji-заголовками (`### Робота з обладнанням 🛠️`,
    `### Веб-панель 💻`, `### Бекенд, API та інтеграції ⚙️`, и т.п.).
- Ссылки на компоненты: из `changelogs/X.md` → `../components/...`;
  из `changelog.md` → `./components/...`.

### Сопутствующие инструкции, которые часто надо обновлять вместе с changelog
- Спецкоманды макросов: `*/docs/components/macros/special-commands.md` (таблица
  конструкций `<sleep>`, `<cr>`, `<prompt>`, `<confirm>`, `<confirm-if>`,
  `<stream_timeout>` и т.д.).
- Регистрация ОНУ: `*/docs/components/onts-registration/registration-macro.md`.

## 2. Где брать изменения (исходный код)

Исходники — **не в этом репозитории**. Они в `~/projects/wildcore/agent/*` —
это контейнер из множества **отдельных** git-репозиториев (НЕ submodules).
Ключевые для changelog:

| Компонент (репо в agent/)      | Что покрывает                                  |
| ------------------------------ | ---------------------------------------------- |
| `switcher-core`                | Работа с оборудованием (OLT/коммутаторы), модули вендоров, консоль/SNMP |
| `wildcore-client-web-panel`    | Веб-панель (фронтенд, Vue)                      |
| `wildcore-agent-api`           | Бэкенд API, компоненты/интеграции, синхронизации |
| `wildcore-agent-install-tool`  | Установка/обновление агента                     |

(в `core/*` лежат другие репозитории — для agent-changelog их не трогаем, если
не сказано иначе.)

## 3. Как определить changelog новой версии

1. **Уточни окно версии у пользователя.** Версия = диапазон дат коммитов
   (например, `0.30 = 2026-01-20 → дата релиза`). Граница важна: одни и те же
   правки при сдвиге окна попадают в разные версии. Не угадывай — спрашивай.
2. **Просканируй все репо `agent/*` за окно**, а не только «очевидные»:
   ```bash
   cd ~/projects/wildcore/agent
   for r in $(find . -maxdepth 2 -name .git -type d | sed 's|/.git||'); do
     n=$(git -C "$r" log --all --since="<FROM>" --until="<TO_EXCLUSIVE>" --oneline | wc -l)
     [ "$n" -gt 0 ] && printf '%-40s %s\n' "${r#./}" "$n"
   done
   ```
3. **Читай реальные diff'ы, а не только сообщения коммитов.** Много коммитов
   называются «Fix»/«Try to fix» — суть видна только в `git show <hash>`.
   Для широкого охвата запускай параллельные Explore-агенты по одному на репо
   (швайбко и не раздувает контекст).
4. **Группируй по темам** и отбирай только то, что заметно пользователю
   (ISP-оператору): новое оборудование/прошивки, новые возможности,
   производительность, исправления реальных проблем. Чисто внутренние рефакторы
   — пропускай.

## 4. Правила формулировок (ВАЖНО — закрытый код)

**Пиши клиентским языком, описывай возможность/пользу, а не реализацию.**

❌ НЕ упоминать:
- имена классов/модулей/слагов (`zte_ont_configuration`, `pon_onts_service_profile`),
  пути файлов, namespace;
- названия фреймворков/библиотек (Vue, Vite, leaflet, workbox, Slim);
- имена репозиториев (`switcher-core`, `wildcore-agent-api`, …);
- внутренние имена методов/эндпоинтов-классов (`GetBoxBy`), env-переменные,
  имена колонок БД.

✅ МОЖНО (это клиентам важно и встречалось в прошлых changelog):
- модели/вендоры оборудования: `ZTE C600`, `C-Data FD1700S (FW3)`, `Huawei`,
  `Juniper QFX5100`, `Mikrotik RouterOS`, `BDcom` и т.п.;
- внешние интегрируемые системы: `UTELS`, `MikBill`, `Userside`,
  `Prometheus/Alertmanager`, «OpenAI-совместимый провайдер»;
- общеупотребимые термины: `PWA`, `OpenAPI/Swagger`, `LACP`, `VLAN`, `SFP`, `ONU/ONT`.

**Всегда уточняй у пользователя исключения по содержанию.** В прошлом просили
убирать: 2FA, AI-ассистента, всё про «боксы»/UTELS. Это решает пользователь —
не додумывай, спрашивай или подтверждай.

**Проверяй текст на «утечки» внутренних имён** перед финалом, напр.:
```bash
grep -rniE 'switcher-core|agent-api|web-panel|\.php|namespace|class [A-Z]|leaflet|workbox|\bvue\b|vite' \
  uk/docs/changelogs/X.md en/docs/changelogs/X.md
```

## 6. Скриншоты для документации (Playwright)

Скриншоты для вики снимаем **автоматически**, гоняя реальную веб-панель в
мобильной эмуляции Chrome через Playwright. Готовый скрипт лежит в репозитории:
`tools/screenshots/mobile-shots.js` (это рабочий файл, **в сайт не попадает**).

### Инструменты в этой среде
- В системе есть `google-chrome` (`/usr/bin/google-chrome`) и глобальный пакет
  `@playwright/mcp`, внутри которого — `playwright-core`. Скрипт сам находит его
  (`loadChromium()`), отдельная установка обычно не нужна. Если нет — `npm i -g playwright`.
- Запуск Node-скриптов и Chrome в этой среде **разрешён** — гонять их можно свободно.

### Где брать данные (демо-стенд, а не прод!)
- **НЕ снимаем с боевого бэкенда** (там реальные клиентские данные: имена, IP,
  серийники, координаты — им не место в публичной вики).
- Демо-стенд: `https://demo.wildcore.tools/`, логин/пароль `admin` / `admin`.
  Данные там безопасные (внизу страниц виден бейдж «DEMO MODE»).

### Как снимать
```bash
WC_BASE=https://demo.wildcore.tools WC_USER=admin WC_PASS=admin \
OUT=/tmp/wcshots node tools/screenshots/mobile-shots.js
# затем просмотреть /tmp/wcshots/*.png и разложить по <lang>/docs/assets/<раздел>/
```
Ключевые приёмы, заложенные в скрипт (пригодятся для новых страниц):
- **Мобильная эмуляция:** контекст с `viewport 390×844`, `isMobile:true`,
  `hasTouch:true`, iPhone-User-Agent. Нижняя навигация рисуется только при
  `innerWidth < 768`, поэтому десктопный контекст её не покажет.
- **Логин:** форма — `input[type=text]` + `input[type=password]` + `button[type=submit]`.
  Токен после входа лежит в `localStorage.auth.key`; API-заголовок — `X-Auth-Key`.
  База API — `/api/v1`.
- **Геолокация для «Объектов поруч»:** выдаём `permissions:['geolocation']` и
  ставим координаты через `ctx.setGeolocation(...)`. Скрипт находит точку с
  реальными объектами, запрашивая `/api/v1/maps/groups`, иначе результатов не будет.
- **Нижняя навигация — важный нюанс:** демо-бэкенд **не сохраняет** настройку
  `bottom_nav_enabled` (фича новее стенда), а перехват ответов не проходит
  (PWA-service worker). Поэтому включаем панель **прямо в живом состоянии Vue**:
  `$auth` доступен через `app.config.globalProperties` (корневой элемент с
  `__vue_app__`). Ставим `auth._auth.user.settings.global.bottom_nav_enabled=true`
  + `bottom_nav_items` и шлём `new CustomEvent('bottom-nav-settings-updated')` —
  рисуется настоящий компонент панели, без подделки разметки.
- **Кропы кнопок/карточек:** снимаем не всю страницу, а элемент
  (`page.locator(...).screenshot()`); устойчивые селекторы берём из исходников
  панели (`switcher`-фронт в `agent/wildcore-client-web-panel/src`).

### Правила по содержимому скриншотов
- Только демо-данные; никаких реальных абонентов/адресов.
- Селекторы и приёмы — это внутрянка: они в `tools/` и в этой заметке, **в `docs/`
  их не тащим** (см. раздел «Правила формулировок»).
- Раскладка: `<lang>/docs/assets/<раздел>/<имя>.png`, синхронно для `uk` и `en`
  (можно один и тот же файл в обе папки). Ссылки из страниц — `../assets/<раздел>/<имя>.png`.

## 7. Git-workflow (особенности окружения)

- **Push только по SSH.** HTTPS-remote `https://github.com/...` в этой среде
  без кредов (логин ввести нельзя). У пользователя `meklis` работает SSH:
  ```bash
  git push git@github.com:wildcore-dms/wiki.git master
  # fetch при необходимости тоже через SSH-URL:
  git fetch git@github.com:wildcore-dms/wiki.git master
  ```
- **Коммить только свои файлы.** В рабочем дереве часто висят посторонние
  незакоммиченные правки (`scrabber.*`, `providers*.csv`, `diagrams/`,
  `_dirs.txt`). Добавляй и коммить строго по pathspec:
  ```bash
  git add -- <свои md-файлы>
  git commit -m "..." -- <те же файлы>
  ```
- **Если push отклонён (fetch first):** удалёнка ушла вперёд (напр., добавили
  версию параллельно). `git fetch` по SSH → `git rebase --autostash FETCH_HEAD`
  (autostash уберёт посторонние правки и вернёт их после). Возможен конфликт в
  `changelog.md` — разрули порядок версий (новая сверху `???+`, ниже свёрнутые
  `???`), `git add`, `GIT_EDITOR=true git rebase --continue`, затем push.
- Трейлер коммита:
  `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>`
